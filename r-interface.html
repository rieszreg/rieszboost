R interface

The R wrapper is an R6 mirror of the Python RieszBooster. R-side and Python-side predictions are bitwise-identical on the same data because the R wrapper delegates every numerical step to Python via reticulate — the R object holds a reference to the underlying Python RieszBooster instance and forwards method calls.

This page is a focused R reference. The R API is covered throughout the rest of the guide via the bilingual tabsets — install in install.qmd, the standard fit pattern in quickstart.qmd, the full estimand inventory in estimands.qmd, and so on.

One-time setup

After installing rieszboost into a Python venv (see install.qmd):

# Option A: env var, set before R starts
Sys.setenv(RETICULATE_PYTHON = "/path/to/rieszboost/.venv/bin/python")

# Option B: explicit, after R has started
use_python_rieszboost("/path/to/rieszboost/.venv/bin/python")

use_python_rieszboost() accepts either a path to a Python interpreter or a path to a virtualenv directory; it forwards to the appropriate reticulate function. It also imports the Python rieszboost module up front so any subsequent ATE(), RieszBooster$new(...), etc. work without per-call import overhead.

API surface (R-side)

Every public Python class and function except the symbolic tracer is exposed in R:

R Python equivalent
ATE(treatment, covariates) rieszboost.ATE(treatment, covariates)
ATT(treatment, covariates) rieszboost.ATT(treatment, covariates)
TSM(level, treatment, covariates) rieszboost.TSM(level, treatment, covariates)
AdditiveShift(delta, treatment, covariates) rieszboost.AdditiveShift(delta, treatment, covariates)
LocalShift(delta, threshold, treatment, covariates) rieszboost.LocalShift(delta, threshold, ...)
StochasticIntervention(samples_key, treatment, covariates) rieszboost.StochasticIntervention(...)
SquaredLoss() rieszboost.SquaredLoss()
KLLoss(max_eta) rieszboost.KLLoss(max_eta)
XGBoostBackend(hessian_floor, gradient_only) rieszboost.XGBoostBackend(...)
SklearnBackend(base_learner_factory) rieszboost.SklearnBackend(...)
RieszBooster$new(estimand, ...) rieszboost.RieszBooster(estimand, ...)
  $fit(df, eval_set=) .fit(X, eval_set=)
  $predict(df) .predict(X)
  $score(df) .score(X)
  $riesz_loss(df) .riesz_loss(X)
  $diagnose(df) .diagnose(X) (returns a list, with $summary as a string)
  $py the underlying Python object (escape hatch)

R6’s $method() syntax invariably maps to Python’s .method(). Integer-typed kwargs (n_estimators, max_depth, random_state, early_stopping_rounds) need an L suffix in R, e.g. n_estimators = 200L. Other kwargs are auto-converted by reticulate.

Verifying parity

The parity claim is not aspirational — it’s tested in CI. Here’s the same fit run from both languages, comparing predictions row-by-row.

  • Python
  • R
import numpy as np, pandas as pd
from rieszboost import RieszBooster, ATE

rng = np.random.default_rng(0)
n = 500
x = rng.uniform(0, 1, n)
a = rng.binomial(1, 1 / (1 + np.exp(-(2 * x - 1))))
df_py = pd.DataFrame({"a": a.astype(float), "x": x})

booster_py = RieszBooster(estimand=ATE(), n_estimators=50, learning_rate=0.05,
                          max_depth=3, random_state=42).fit(df_py)
alpha_py = booster_py.predict(df_py)
alpha_py[:5]
array([ 1.4076698,  1.4764098,  1.6470399, -1.1931754,  1.1374395],
      dtype=float32)
set.seed(0)
n  <- 500
x  <- runif(n)
a  <- rbinom(n, 1, 1 / (1 + exp(-(2 * x - 1))))
df_r <- data.frame(a = as.numeric(a), x = x)

booster_r <- RieszBooster$new(
  estimand = ATE(), n_estimators = 50L, learning_rate = 0.05,
  max_depth = 3L, random_state = 42L
)
booster_r$fit(df_r)
alpha_r <- booster_r$predict(df_r)
head(alpha_r, 5)
[1]  1.212691 -1.263393  1.486148 -1.365574  1.212691

(The two RNGs are different — Python’s default_rng(0) and R’s set.seed(0) give different x and a columns, so alpha_py and alpha_r are not directly comparable here. To verify parity end-to-end, generate the data in one language, pass it to the other via reticulate, and compare predictions on the same df. The R package’s test suite includes exactly that test.)

Calling Python-only features from R

The LinearForm tracer is Python-only — for now, custom estimands (those needing the tracer) require defining m() in Python. From R you can still drive a custom Python m() end-to-end via reticulate:

# Define a custom m in Python (e.g. in a script you `source` from R)
reticulate::py_run_string("
def m_my_thing(z, alpha):
    return 2.0 * (alpha(a=2, x=z['x']) - alpha(a=0, x=z['x']))

from rieszboost import Estimand
my_estimand = Estimand(feature_keys=('a', 'x'), m=m_my_thing, name='MyThing')
")

est <- reticulate::py$my_estimand
booster <- RieszBooster$new(estimand = est, n_estimators = 100L, max_depth = 3L)
booster$fit(df_r)
booster$predict(df_r)[1:5]

The same pattern works for custom LossSpec subclasses: define the class in Python, grab the instance via reticulate::py$..., pass to RieszBooster$new(loss = ...).

Escape hatch

Every R6 method ultimately calls into the underlying Python object. If something isn’t yet wrapped, you can always reach through:

booster <- RieszBooster$new(estimand = ATE(), n_estimators = 50L)
booster$fit(df_r)

# These exist on the Python side:
reticulate::py_to_r(booster$py$best_iteration_)
reticulate::py_to_r(booster$py$best_score_)
reticulate::py_to_r(booster$py$base_score_)

The $py field is the underlying Python object; reticulate handles the conversion both ways. Use this for occasional access to attributes the R6 wrapper doesn’t expose; if you find yourself using it constantly, file an issue and we’ll add an R-side accessor.

What’s not in R yet

Feature Workaround
Custom m() for new estimands (the symbolic tracer is Python-only) Define m() in Python via reticulate::py_run_string, then pass the resulting Estimand to RieszBooster$new(estimand=...). See above.
Custom LossSpec subclasses Same pattern: subclass in Python via reticulate::py_run_string, pass via loss=.

Everything else — every built-in estimand, both backends, both losses, the full RieszBooster API including cross_val_predict via reticulate (see sklearn-integration.qmd) — is first-class in R.
